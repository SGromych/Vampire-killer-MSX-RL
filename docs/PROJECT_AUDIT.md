# Аудит проекта RL_msx и рекомендации

**Назначение:** обзор состояния проекта, оценка «рефакторинг vs с нуля», план по исправлению env 1 (кнопки не нажимаются) и при необходимости — краткое ТЗ по игре и архитектуре.

Связанные документы: `docs/MULTI_ENV_DEBUG_SESSION.md`, `docs/CONTEXT.md`, `docs/MODULES_AND_FLAGS.md`, `docs/CAPTURE.md`, `docs/TRAINING.md`.

## 0. Рефакторинг (выполнено)

- **Детектор смерти:** добавлен `death_warmup_steps` (по умолчанию 50): в первых N шагах смерть не объявляется (защита от артефактов кадра/переходов). STAGE 00 override сделан безусловным: при step≤50, stage=0 и stage_conf≥0.5 смерть считается ложной и не прерывает эпизод (раньше срабатывало только при debug + debug_force_action).
- **Диагностика вынесена в модуль:** `msx_env/env_diagnostics.py` — реестр ресурсов (`ENV_RESOURCE_REGISTRY`, `clear_env_resource_registry`, `register_and_assert_resources`), вывод блоков ENV RESET и TERMINATION только при debug (`print_reset_block`, `print_step_control`, `print_termination_block`). В `env.py` остаётся только вызов этих функций.
- **train_ppo:** добавлен флаг `--no-reset-handshake` — при num_envs>1 отключает reset handshake (для проверки, не мешает ли он работе кнопок во втором env).
- **Импорты:** `clear_env_resource_registry` импортируется из `msx_env.env_diagnostics` (в train_ppo).

## 1. Текущее состояние проекта

### 1.1. Что уже работает

- **Один env (num_envs=1):** управление openMSX по файлам (commands.tcl / reply.txt), захват PNG, BC и PPO обучение, тесты политик.
- **Цепочка:** запись демо → BC → PPO с инициализацией из BC; модульная система наград (reward/), HUD-парсер, детектор смерти с гистерезисом.
- **Документация:** CONTEXT, PPO_MODEL, MODULES_AND_FLAGS, REWARD, CAPTURE, TRAINING, MULTI_ENV_DEBUG_SESSION — есть ясная картина решений и флагов.
- **Изоляция по workdir:** при num_envs>1 у каждого env свой `workdir = tmp_root/rank`, свой процесс openMSX, свой bootstrap.tcl и commands.tcl; коллизия по window capture при нескольких env устранена (fallback на png при отсутствии --window-rects-json).

### 1.2. Известная проблема: env 1 (второй инстанс)

**Симптом:** при `num_envs=2` в env 0 кнопки работают (герой идёт вправо при RIGHT), в env 1 кнопки не нажимаются — игра во втором окне фактически без ввода.

**Гипотезы из MULTI_ENV_DEBUG_SESSION:**

- Команды не доходят до второго процесса openMSX, **или**
- Второй процесс не читает свой `commands.tcl` (не тот workdir, блокировка файла, порядок/тайминг опроса).

**Что уже проверено:**

- Workdir и PID у env 0 и env 1 разные; реестр ресурсов при debug не показывает коллизий.
- Управление идёт через свой канал: каждый env держит свой `OpenMSXFileControl(workdir=tmp_root/rank)`; в `step()` вызывается `_apply_action(self._emu, action)` — запись в свой `commands.tcl`, свой процесс должен опрашивать свой файл.

**Вероятные причины (по приоритету):**

1. **Тайминг/порядок запуска:** второй процесс стартует сразу после первого; возможна гонка по портам/файлам или «неготовность» второго инстанса к приёму команд (несмотря на _wait_boot и задержку 3 с между reset в train_ppo).
2. **Поведение openMSX при двух процессах:** неочевидная привязка к порту/конфигу; второй процесс может работать в урезанном режиме (например, не обрабатывать скрипты так же, как первый).
3. **Файловый канал под Windows:** блокировка/кэш при записи или чтении `commands.tcl`/reply.txt вторым процессом (например, антивирус или общий диск).
4. **Путь в bootstrap.tcl:** теоретически ошибка в абсолютном пути для второго workdir (например, кодирование, пробелы) — маловероятно, т.к. путь строится так же, как для env 0.

## 2. Рекомендации: починить env 1

### 2.1. Диагностика (сделать в первую очередь)

1. **Проверить потребление commands.tcl во втором workdir**
   - После `envs[1].step(RIGHT)` проверить, что в `runs/tmp/1/commands.tcl` файл был прочитан и обнулён (как делает bootstrap после `source`). Если после шага файл по-прежнему содержит Tcl — второй процесс не опрашивает этот файл или опрашивает другой.
2. **Проверить ответ второго процесса**
   - Временно после шага env 1: отправить простую команду с уникальным RID только для env 1 (например ping), дождаться ответа в `runs/tmp/1/reply.txt`. Если ответ приходит — канал работает, проблема в keymatrix/привязке клавиш. Если не приходит — канал (файл/процесс) не работает.
3. **Логи второго openMSX**
   - Просмотреть `runs/tmp/1/openmsx_stdout.log` (или отдельный лог по instance_id, если задан log_dir) на предмет ошибок при загрузке ROM, bootstrap или при выполнении keymatrix.
4. **bootstrap_status.txt для env 1**
   - В цикле опроса bootstrap пишет `cmd_exists=... cmd_size=...`. Убедиться, что для env 1 после записи команд размер становится >0, а после обработки — снова 0.

### 2.2. Исправления по результатам диагностики

- **Если второй процесс не читает commands.tcl:** увеличить задержку между запуском первого и второго инстанса; добавить явную проверку «второй процесс ответил на ping» перед началом rollout; проверить документацию openMSX по запуску нескольких копий (порты, конфиги).
- **Если читает, но клавиши не срабатывают:** проверить, что keymatrixdown/up выполняются в контексте именно второго процесса (не общий глобальный ввод); при необходимости рассмотреть управление через сокет/pipe по документации openMSX для однозначной привязки канала к инстансу.
- **Жизненный цикл:** при старте train_ppo при num_envs>1 проверять наличие «забытых» процессов openMSX (по PID или lock-файлу в runs/tmp) и выводить предупреждение или опцию «убить старые инстансы» (как в п.5 рекомендаций MULTI_ENV_DEBUG_SESSION).

### 2.3. Интеграционный тест

- Добавить тест: 2 env, по 10–20 шагов с принудительным RIGHT, проверка что:
  - у обоих env `ep_steps` растёт и при необходимости `room_hash` или frame_hash различаются;
  - в STAGE 00 нет ложных смертей (или они переопределены по текущей логике).

## 3. Рефакторинг vs проект с нуля

### 3.1. Вывод: целесообразен рефакторинг и исправление ошибок, а не полная переписка

**Почему не делать проект с нуля:**

1. **Рабочий объём уже большой и специфичный:** интеграция с openMSX (Tcl, keymatrix, screenshot, bootstrap), несколько бэкендов захвата (png/single/window), калибровка окна, модульная система наград, BC + PPO с рекуррентом, демозапись и валидация. Повторная реализация того же с нуля займёт много времени и снова пройдёт через те же подводные камни (смерть по одному кадру, коллизия window при num_envs>1, чтение PNG и т.д.).
2. **Архитектура в целом правильная:** чёткое разделение env / openmsx_bridge / capture / reward / make_env; конфиги и флаги задокументированы; изоляция по workdir при multi-env заложена изначально.
3. **Проблема env 1 локализована:** это не провал архитектуры, а скорее тайминг, канал управления или особенность второго инстанса openMSX. Исправление через диагностику и точечные правки реалистично.
4. **Риски переписывания:** легко потерять уже исправленные вещи (гистерезис смерти, BytesIO+retries для PNG, fallback window→png, STAGE 00 override, per-env статистика) и снова ввести регрессии.

**Когда имело бы смысл переписать с нуля:**

- Если бы выяснилось, что файловый протокол в принципе не позволяет надёжно работать двум инстансам openMSX и единственный путь — сокет/pipe и полная смена протокола; и то разумнее сначала попробовать заменить только канал управления в текущем коде.
- Если бы требования к игре/наградам/наблюдениям кардинально поменялись (другая игра, другой эмулятор) — тогда да, новый проект.

### 3.2. Что оставить и что рефакторить (в рамках текущего кода)

**Оставить как есть (уже исправлено и задокументировано):**

- Гистерезис детектора смерти; чтение скриншота через read_bytes + BytesIO и повторные попытки; изоляция workdir и fallback window→png при num_envs>1 без per-env rects; задержка 3 с между запусками инстансов; проверка процесса после _wait_boot; смягчение STAGE 00 (warning + override); per-env диагностика при --debug.

**Рефакторинг (по желанию, не блокирующий env 1):**

- Вынести диагностику (ENV RESET, TERMINATION, per-env stats) в отдельный модуль/класс (например `EnvDiagnostics`) и включать только при debug.
- Упростить вывод ENV RESET в не-debug режиме (одна строка или отключить).
- Реестр ресурсов и проверки коллизий оставить только при debug или num_envs>1.
- Reset handshake при num_envs>1 сделать опциональным флагом, если мешает старту.

**Не откатывать:** логику смерти (гистерезис), чтение скриншота (BytesIO + retries), изоляцию workdir и fallback window→png.

## 4. ТЗ по игре и архитектуре (краткое)

Имеет смысл зафиксировать краткое ТЗ не для переписывания проекта, а как **референс при рефакторинге и отладке**, чтобы не терять контекст и не ломать принятые решения.

### 4.1. Игра и цель

- **Игра:** Vampire Killer (Konami, MSX2), эмулятор openMSX.
- **Цель агента:** двигаться по уровню, подбирать призы (сердечки, оружие, ключи), найти выход; врагов бить по необходимости, не застревать в бою.
- **Цепочка обучения:** демо → BC (имитация) → PPO (RL по награде); BC остаётся для теста через test_policy; PPO — через test_ppo.

### 4.2. Управление и наблюдения

- **Управление:** file-based (Windows): Python пишет `commands.tcl`, bootstrap.tcl в workdir опрашивает и выполняет; ответ в `reply.txt`. Клавиши — keymatrix (keymatrixdown/up) и `type` (SPACE/Enter для меню).
- **Действия:** 10 дискретных (NOOP, RIGHT, LEFT, UP, DOWN, ATTACK, RIGHT_JUMP, LEFT_JUMP, RIGHT_JUMP_ATTACK, LEFT_JUMP_ATTACK).
- **Наблюдение:** 84×84 grayscale, frame stacking 4 кадра; опционально reward/HUD из того же кадра (без второго чтения PNG, если уже передаётся rgb).

### 4.3. Модули (роли)

- **openmsx_bridge:** один процесс openMSX на один workdir; запуск с -script bootstrap.tcl; запись commands.tcl (атомарно tmp→replace), ожидание ответа в reply.txt; screenshot, keymatrix, type.
- **msx_env.env:** VampireKillerEnv — reset (интро/продолжение), step(action) — применение действия, захват кадра, obs, reward, terminated/truncated, info. Использует bridge и capture.
- **msx_env.capture:** абстракция захвата кадра (png/single/window); один источник на env; grab() → RGB.
- **msx_env.make_env:** при num_envs>1 — уникальный workdir=tmp_root/rank, instance_id=rank; при window — per_env_window (crop/title по рангу).
- **msx_env.reward:** модульная политика наград (step, death, pickup, novelty, stuck и т.д.); конфиг RewardConfig; при отсутствии — legacy (HUD + death в env, step_penalty в train_ppo).
- **Детектор смерти:** по ROI полоски жизни (life_bar); гистерезис (два подряд кадра life<0.15); в STAGE 00 при первых 50 шагах при debug_force_action — только предупреждение, без terminated.

### 4.4. Multi-env

- Каждый env: свой workdir, свой процесс openMSX, свой commands.tcl/reply.txt. Захват: при num_envs>1 и window без per-env rects — принудительно png (file). Между запусками инстансов — задержка (например 3 с). При необходимости — проверка «второй процесс ответил на ping» и опция очистки старых процессов перед стартом.

Это ТЗ можно хранить в `docs/CONTEXT.md` (расширить) или в отдельном `docs/SPEC.md` и использовать как чеклист при изменениях.

## 5. План действий (кратко)

1. **Исправить env 1:** провести диагностику по п. 2.1 (потребление commands.tcl, ping для env 1, логи, bootstrap_status), затем по результатам применить п. 2.2; добавить интеграционный тест по п. 2.3.
2. **Рефакторинг (по мере необходимости):** вынести диагностику в отдельный модуль; оставить реестр и детальный вывод только в debug; при желании сделать reset handshake опциональным.
3. **ТЗ/архитектура:** оформить/дополнить краткое ТЗ (игра, цель, управление, модули, multi-env) в docs и опираться на него при доработках.
4. **Не переписывать проект с нуля** — продолжать рефакторинг и исправление ошибок в текущей кодовой базе.
