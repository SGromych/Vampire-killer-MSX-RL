# Бенчмарк: CNN vs CNN+LSTM (режим recurrent)

Сравнение трёх коротких экспериментов для проверки, что память (LSTM) не ломает обучение и при необходимости улучшает исследование.

## Эксперименты

Запускать с малым числом обновлений (например 5–10) и `--num-envs 2` для воспроизводимости. ROM и env должны быть доступны.

### 1) CNN only (baseline)

Без LSTM, frame stack 4 (как по умолчанию).

```bash
python train_ppo.py --run-name bench_cnn --epochs 10 --num-envs 2
```

### 2) CNN + frame stack (тот же baseline)

Frame stack уже 4 по умолчанию; при необходимости явно проверить, что стопка не отключена. То же, что эксперимент 1.

### 3) CNN + LSTM

Включён рекуррентный режим, LSTM после encoder.

```bash
python train_ppo.py --run-name bench_lstm --epochs 10 --num-envs 2 --recurrent --lstm-hidden-size 256
```

## Метрики для сравнения

После прогона смотреть в консоль и/или `metrics.csv` (если есть):

- **unique_rooms** (mean за rollout) — ожидание: при LSTM не хуже или выше за счёт памяти.
- **stuck_events** — число событий stuck за rollout; желательно не расти.
- **deaths** — число смертей за rollout.
- **entropy** — средняя энтропия политики; при LSTM может отличаться.
- **h_norm** (только при `--recurrent`) — средняя норма скрытого состояния LSTM.

## Критерии приёмки

- В режиме без `--recurrent` поведение и метрики как до добавления LSTM (нет регрессии).
- При `--num-envs 2` и `--recurrent` обучение идёт без ошибок, скрытые состояния сбрасываются на done.
- При равных условиях (epochs, num_envs) LSTM не даёт сильного проигрыша по unique_rooms/stuck/deaths; при успешной настройке возможен выигрыш по исследованию.

## CLI (recurrent)

| Флаг | По умолчанию | Описание |
|------|--------------|----------|
| `--recurrent` | False | Включить LSTM после encoder. |
| `--lstm-hidden-size` | 256 | Размер скрытого состояния LSTM. |
| `--sequence-length` | 0 | Зарезервировано для диагностики/усечённого BPTT (пока не используется). |
