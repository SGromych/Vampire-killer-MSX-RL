# Анализ: рефакторинг capture + stepping для RL env

## Ответы на явные вопросы

### 1. Нужно ли менять существующие датасеты?

**Нет**, при условии что формат наблюдения остаётся тем же:

- **Текущий формат:** `obs` — массив `(H, W)` = (84, 84), `dtype` uint8, grayscale (один кадр на шаг). В `data.npz`: `obs`, `next_obs`, `actions`, `rewards`, `dones`, `timestamps`. В манифесте: `obs_shape`, `obs_dtype`, `action_set_version`.
- **Обязательно сохраняем:** тот же `frame_size=(84, 84)`, та же цепочка: сырой экран → resize → grayscale → uint8. Источник кадра (PNG с диска, один файл, окно) не должен менять итоговый вид массива.

Итог: **датасеты не трогаем**, меняем только способ захвата и частоту шагов.

---

### 2. Остаются ли старые BC-датасеты валидными?

**Да.** BC загружает `load_demo_run()` → `obs` (N, 84, 84), затем `obs.astype(np.float32) / 255.0` и frame stacking (4 кадра). Модели ожидают вход (B, C, 84, 84), C=4. Если env по-прежнему отдаёт (84, 84) uint8 и препроцессинг в train_bc/train_ppo не меняется (те же resize/scale), старые демо и чекпоинты остаются валидными.

---

### 3. Остаётся ли препроцессинг наблюдений идентичным?

**Да, должен остаться:**

- Вход: кадр экрана (как сейчас — PNG 256×192 или область окна того же содержимого).
- Цепочка: resize до `frame_size` (84, 84) → convert to L (grayscale) → `np.array(..., dtype=np.uint8)`.
- Выход: `(84, 84) uint8`.

Если в будущем появится другой кроп/разрешение (например, только игровая область при window capture), тогда нужно либо:

- **(a)** оставить старый формат и ввести версию датасета (например `obs_version=1` / `obs_version=2`), либо  
- **(b)** мигрировать датасеты (перезаписать obs через новый пайплайн), либо  
- **(c)** поддерживать оба формата по флагу (сложнее).

**В рамках Phase 1–2:** источник кадра меняется (файл/окно), но геометрия и препроцессинг — те же 84×84 grayscale → **миграция не нужна, обратная совместимость сохраняется.**

## Где сейчас уходит время (блокирующие вызовы)

Текущий шаг `step()`:

1. **`_apply_action(emu, action)`**  
   - keyup/keydown → `_send_tcl_with_replyproc` → `_wait_reply_rid`.  
   - 1–2 round-trip по poll (poll_ms=15): запись в `commands.tcl`, ожидание ответа в `reply.txt`.

2. **`_grab_obs(emu)`**  
   - `emu.screenshot("step_frame.png")`: Tcl `screenshot <path>` → openMSX пишет PNG на диск → пишет reply.  
   - **Блокировка:** `_wait_reply_rid` (до timeout_s=3), т.е. ждём завершения записи PNG и ответа.  
   - `PIL.Image.open(img_path).convert("RGB")` — **второй тяжёлый шаг:** чтение только что записанного файла с диска + декод PNG.  
   - `img.resize((84,84)).convert("L")` и `np.array` — относительно дёшево.

3. **HUD reward**  
   - `parse_hud(Path(cfg.workdir) / "step_frame.png")` → снова **`Image.open(step_frame.png)`** — тот же файл читается второй раз за шаг.

4. **terminated_on_death**  
   - `get_life_estimate(obs)` — по уже полученному массиву, без доп. I/O.

Итого на один шаг:

- 1× (или 2×) round-trip на действие (Tcl).
- 1× round-trip на screenshot (Tcl + запись PNG openMSX).
- 2× чтение одного и того же PNG с диска (grab_obs + parse_hud).

Главные узкие места: **запись PNG эмулятором**, **ожидание ответа по файлам**, **два чтения одного PNG** с диска.

## Что даст наибольший выигрыш

| Изменение | Эффект |
|-----------|--------|
| **Один файл (overwrite)** | Меньше операций с ФС (один и тот же файл), чуть меньше лаг при записи. Умеренный выигрыш. |
| **Убрать двойное чтение** | Передавать в `parse_hud` уже загруженное изображение (или obs после resize), а не путь. Один раз прочитали — и для obs, и для HUD. Заметное сокращение I/O на шаг. |
| **Window capture (mss/dxcam)** | Убрать запись PNG в openMSX и чтение с диска. Захват из буфера окна → тот же resize/grayscale. Самый большой выигрыш по времени шага. |
| **Action repeat** | Меньше вызовов capture на «логический» шаг: N шагов эмулятора, 1 захват. Прямо увеличивает «шаги в минуту» для алгоритма при той же нагрузке на I/O. |
| **Decision FPS** | Стабильная частота решений (например 10–15 Hz), предсказуемое время и меньше лишних шагов. |

Рекомендуемый порядок внедрения по выигрышу: убрать двойное чтение → single-file overwrite → action_repeat + decision_fps → window capture.

## Совместимость датасетов и чекпоинтов

- **Датасеты (data.npz + manifest):** остаются валидными, если не менять `obs_shape`/`obs_dtype` и препроцессинг (84×84 uint8). Версионирование (например `dataset_version` или `obs_version`) имеет смысл добавить, если позже появится альтернативный формат obs.
- **BC-чекпоинты:** зависят только от архитектуры и входа (B, C, 84, 84). Формат входа не меняется → чекпоинты остаются валидными.
- **PPO-чекпоинты:** то же самое; инициализация из BC и формат rollout не зависят от способа захвата кадра.

## Дополнительно: готовность к параллельным env (Phase 3)

Сейчас:

- `workdir` задаётся в `EnvConfig` и передаётся в `OpenMSXFileControl`; путь к `step_frame.png` = `workdir / "step_frame.png"`.
- Глобального состояния в env нет; статичные имена файлов привязаны к `self.cfg.workdir`.

Для SubprocVecEnv нужно:

- Каждый инстанс env со своим `workdir` (уже поддерживается через конфиг).
- Каждый инстанс со своим backend захвата и, при window capture, своей областью (или одним окном на процесс).
- Фабрика вида `lambda: VampireKillerEnv(EnvConfig(..., workdir=unique_temp_dir))` без общих путей и без статичных глобальных переменных.

Текущая конструкция уже близка к этому; рефакторинг capture и параметризация `workdir`/region доведут до полной готовности.

## Phase 1 — Реализовано

- **action_repeat** (default=1): N внутренних шагов, 1 захват в конце цикла.
- **decision_fps**: фиксированная частота решений; `step_duration = 1/decision_fps` между внутренними шагами.
- **capture_backend** (png | single): модульный бэкенд в `msx_env/capture.py`; `grab()` возвращает RGB, один вызов — без двойного чтения PNG.
- **parse_hud(rgb)** вместо `parse_hud(path)` — устранено двойное чтение файла.
- **Performance logging**: каждые 100 шагов — capture_time, preprocessing_time, steps_per_min.
- **CLI**: `--action-repeat`, `--decision-fps`, `--capture-backend` в train_ppo, record_demo, test_ppo, test_policy.
- **benchmark_env.py**: `python benchmark_env.py --backend png|single` — 1000 шагов, steps/sec, steps/min.

## Итог

- **Датасеты менять не нужно;** старые BC-датасеты и препроцессинг остаются валидными при сохранении формата (84, 84) uint8.
- **Наибольший выигрыш:** убрать двойное чтение PNG, затем window capture; затем action_repeat и фиксированный decision FPS; single-file — умеренный выигрыш.
- **Обратная совместимость:** сохраняется, если не менять форму/тип obs и цепочку resize → grayscale → uint8. При любом изменении кропа/размера в будущем — явно ввести версию датасета и, при необходимости, стратегию миграции (a/b/c выше).
