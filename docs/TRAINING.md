# PPO-обучение Vampire Killer: аудит, стабильность, эксперименты

Документ описывает диагностику обучения PPO, рекомендуемые улучшения стабильности, систему экспериментов и отладку. Связанные документы: **`docs/REWARD.md`** (награды), **`docs/MODULES_AND_FLAGS.md`** (флаги train_ppo), **`docs/CONTEXT.md`** (архитектура).

---

## 1. Архитектура обучения

- **Цепочка:** Rollout (N env × rollout_steps шагов) → GAE по траекториям каждого env → нормализация advantage → несколько эпох PPO по мини-батчам → сохранение чекпоинта и метрик.
- **Модель:** ActorCritic (shared encoder + actor/critic heads), см. `msx_env/ppo_model.py`. Поддержка инициализации из BC (`init_from_bc`).
- **Окружение:** `VampireKillerEnv` с модульной политикой наград (`RewardPolicy`), см. `docs/REWARD.md`.
- **Мульти-инстанс:** при `--num-envs > 1` у каждого env свой workdir (`tmp_root/0`, `tmp_root/1`), GAE считается по траекториям отдельно, rollout собирается по очереди (env0, env1, env0, …). Подробнее: `docs/MODULES_AND_FLAGS.md`, секция «Мульти-инстанс».
- **Захват кадра:** бэкенды png/single/window через `msx_env/capture.py`; калибровка и бенчмарк — `docs/CAPTURE.md`.

## 2. Аудит и диагностика (STEP 1)

Перед изменениями проведён анализ кода и типичных точек отказа. Исторические логи (TensorBoard/CSV) в проекте не накапливались, поэтому диагностика опирается на то, что **не логировалось** и где возможны узкие места.

### 2.1. Что раньше не логировалось

- **Value loss, policy loss, entropy** — не выводились; нельзя было судить, учится ли критика и не коллапсирует ли политика.
- **Explained variance** — не считалась; неясно, насколько value предсказывает returns.
- **KL (approx)** — не логировался; сложно подобрать clip_eps и число PPO-эпох.
- **Разбивка наград** — по шагам и эпизодам печатался только отчёт в конце эпизода в консоль, агрегация по rollout для анализа не велась.
- **Эпизодные статистики** — unique_rooms, частота смертей, stuck-события не агрегировались по обновлениям.

### 2.2. Типичные узкие места (топ-3)

1. **Слабый или шумный reward signal**  
   Step penalty (−0.001) и редкие события (pickup, novelty) при малом числе шагов в rollout дают мало информации; death (−1.0) может доминировать. Без нормализации или масштабирования градиенты и value могут быть нестабильны.

2. **Коллапс энтропии политики**  
   При фиксированном малом `entropy_coef` (0.01) политика может быстро стать почти детерминированной; exploration падает, unique_rooms перестают расти, обучение «залипает».

3. **Расходимость критика**  
   При неверном масштабе наград или большом lr value loss растёт, explained variance уходит в минус; политика обучается по зашумлённому advantage.

### 2.3. Структурированная сводка диагностики

| Область | Сигнал | Дисперсия | Коллапс | Критик | Масштаб наград |
|--------|--------|-----------|---------|--------|----------------|
| Reward | Редкие события (pickup, novelty) дают разреженный сигнал | Высокая из-за длины траекторий и мульти-env | — | — | Step/death/pickup в одном порядке величины; возможна нормализация |
| Policy | До введения логов неизвестно | — | Риск при низкой энтропии | — | — |
| Value | — | — | — | Без expl. var. не видно | Рекомендуется следить за value_loss и explained_var |
| Эпизоды | unique_rooms, deaths, stuck не агрегировались | — | Застревание без роста комнат | — | — |

**Итог:** в коде добавлено логирование policy_loss, value_loss, entropy, approx_kl, explained_var, агрегация reward components и эпизодных статистик (unique_rooms, deaths, stuck), а также guardrails (см. ниже).

## 3. Предлагаемые улучшения стабильности (STEP 2)

### A) Стабильность PPO

- **Learning rate:** по умолчанию 3e-4; для более стабильного обучения можно 2.5e-4 (см. `configs/ppo_default.json`).
- **Entropy coefficient:** 0.01; при быстром падении энтропии увеличить до 0.02 или ввести schedule (уменьшение со временем).
- **Clip range:** 0.2 — стандарт; при больших обновлениях (высокий KL) можно уменьшить до 0.15.
- **Value loss coefficient:** 0.5; при доминировании value loss над policy — уменьшить (0.25).
- **Rollout vs num_envs:** rollout_steps × num_envs = объём данных за обновление; при малом числе env лучше увеличить rollout_steps (например 256).
- **Batch size:** 64; при большом rollout можно 128 для более стабильных градиентов.

### B) Масштабирование наград

- Нормализовать величину reward (например running mean/std по шагам или эпизодам) — опционально, при внедрении не ломать BC/датасеты.
- Баланс novelty vs penalties: в v1 novelty_reward=0.2, stuck_penalty=−0.2; при доминировании штрафов уменьшить их по модулю или увеличить novelty.
- Детекция эксплуатации: уже есть pickup_cooldown_steps, pingpong_penalty; при необходимости усилить или добавить cap на частоту одних и тех же событий.

### C) Исследование

- **Entropy schedule:** линейное снижение entropy_coef от 0.02 к 0.01 за N обновлений (пока не реализовано; при желании добавить в train_ppo).
- **Entropy floor:** при энтропии ниже порога (по умолчанию 0.3) выводится предупреждение и рекомендуется поднять --entropy-coef.
- Curiosity / intrinsic reward: при необходимости можно добавить отдельный модуль; сейчас exploration стимулируется novelty по комнатам.

### D) Структура эпизода

- **Curriculum:** фокус на уровне 1 (уже по умолчанию); при переходе на другие уровни — отдельные конфиги.
- **Короткие эпизоды на старте:** уменьшить max_episode_steps в первых обновлениях (опционально, через конфиг или отдельный скрипт).
- **Авто-сброс при застревании:** stuck_truncate=True в RewardConfig уже обрезает эпизод при детекции stuck.

## 4. Система экспериментов (STEP 3)

### 4.1. Конфиг эксперимента (JSON)

- Файл конфига (например `configs/ppo_default.json`) задаёт: `lr`, `entropy_coef`, `value_loss_coef`, `clip_eps`, `rollout_steps`, `ppo_epochs`, `batch_size`. Все ключи соответствуют аргументам CLI.
- Запуск: `python train_ppo.py --config configs/ppo_default.json --run-name exp_01`
- CLI переопределяет значения из конфига.

### 4.2. Конфиг наград (JSON)

- Формат — поля `RewardConfig` (см. `msx_env/reward/config.py`). Пример: `step_penalty`, `death_penalty`, `pickup_weapon`, `novelty_reward`, `stuck_penalty` и т.д.
- Запуск: `python train_ppo.py --reward-config configs/reward_v2.json`
- Если файл не указан, используется `default_v1_config()`.

### 4.3. Имя запуска и каталог логов

- `--run-name exp_X` — имя эксперимента; логи и снимок конфига пишутся в run directory (см. ниже).
- **Run directory:** при запуске через супервизор с `use_runs_dir: true` — `runs/<YYYYMMDD>_<HHMMSS>_<git>_<run_name>/`; иначе `<log-dir>/<run_name>/` (по умолчанию `<checkpoint-dir>/<run_name>/`).
- `--log-dir` — каталог логов при отсутствии `--run-dir` (по умолчанию совпадает с `--checkpoint-dir`).

### 4.4. Логируемые метрики

- **Через консоль (каждые 5 обновлений и последнее):** steps_per_sec, sample_throughput, policy_loss, value_loss, entropy, approx_kl, explained_var, reward_mean, ep_return_mean, unique_rooms_mean, deaths, stuck_events.
- **В CSV** (`<run-dir>/metrics.csv`): те же величины плюс разбивка наград по компонентам (step, pickup, death, novelty, pingpong, stuck) и расширенные эпизодные метрики с суффиксом `_ep`
  (см. **`docs/EPISODE_METRICS_FIX.md`** для формального списка и семантики). В некоторых run'ах файл может называться `metrics1.csv` — при поиске «сегодняшнего» прогона проверяйте оба варианта.
- **Снимок конфига:** `<run-dir>/config_snapshot.json` — run_name, гиперпараметры PPO и полный reward_config на момент старта.
- **Run directory:** при `use_runs_dir: true` в `configs/night_training.json` каталог имеет вид `runs/<YYYYMMDD>_<HHMMSS>_<git>_<run_name>/`; актуальный ночной прогон — папка с самой свежей датой/временем изменения файлов (см. **`docs/PROJECT_OVERVIEW.md`** § Run directory).

### 4.5. Примеры команд

```bash
# Базовый запуск с именем эксперимента
python train_ppo.py --run-name exp_01 --epochs 100

# С конфигом PPO и отдельным конфигом наград
python train_ppo.py --config configs/ppo_default.json --reward-config configs/reward_v2.json --run-name exp_02

# Подбор lr и энтропии
python train_ppo.py --run-name exp_lr25 --lr 2.5e-4 --entropy-coef 0.01 --num-envs 4

# Несколько env и свой log-dir
python train_ppo.py --num-envs 4 --run-name exp_4env --log-dir runs/ppo
```

## 5. Guardrails (STEP 4)

При обнаружении проблем выводятся предупреждения (без остановки обучения):

- **NaN в loss/entropy** — проверка после каждого обновления; рекомендация: проверить lr и масштаб наград.
- **Низкая энтропия** — если средняя энтропия по батчам ниже `--entropy-floor` (по умолчанию 0.3): риск коллапса политики; рассмотреть увеличение `--entropy-coef`.
- **Расходимость критика** — если value_loss > 5 или explained_var < −0.5: предупреждение о возможной расходимости; проверить масштаб наград и lr.
- **Застревание обучения** — если за последние `--stuck-updates` (по умолчанию 20) обновлений unique_rooms не растут (и остаются ≤ 1): предупреждение; проверить награды и исследование.

Опциональная автоматическая подстройка энтропии (например увеличение entropy_coef при срабатывании порога) может быть добавлена отдельно; сейчас только вывод предупреждений.

## 6. Ночной режим (quiet, логи, summary)

- **По умолчанию** включён тихий режим: в консоль не выводятся сообщения «Update N env return», «backup saved», «saved to»; отчёт эпизода в env отключён. Подробный вывод: `--no-quiet`.
- **Логи:** Python — `train.log` в каталоге run'а (`<log-dir>/<run_name>/train.log`); openMSX — по одному файлу на инстанс: `openmsx_0.log`, `openmsx_1.log` в том же каталоге. В консоль при quiet выводятся только предупреждения (guardrails) и **компактный summary**.
- **Summary:** каждые `--summary-every` обновлений (по умолчанию 10) или не реже чем раз в `--summary-interval-sec` секунд (по умолчанию 60) печатается один блок: uptime, total_steps, steps/s, **Training health** (см. ниже), rollouts, policy_loss, value_loss, entropy, unique_rooms, stage_mean, deaths, stuck, путь к last.pt.
- **Как понять, что обучение реально идёт:** в каждом summary выводится строка **Training health:** `updates↑ | policy_loss=changing/stable | value_loss=changing/stable | entropy=↓/→/↑ | unique_rooms=↑/→/↓`. Ожидается: updates растут, policy_loss и value_loss меняются между summary, entropy постепенно падает (↓), unique_rooms может расти (↑). Если **steps/s=0**, в блоке появляется предупреждение «rollout может не собираться» — значит, шаги не набираются (env/openMSX зависает или не отдаёт кадры).
- **Кадр:** во время обучения в консоль не идёт поток «screenshot»; захват по-прежнему пишет в workdir один файл (step_frame.png) для чтения кадра, без дампа отладочных изображений.

## 7. Проверка скорости перед ночным запуском (dry run)

Перед длительным ночным прогоном полезно замерить **фактическую скорость** на вашей машине (2 минуты), чтобы задать реалистичный `max_total_steps` или оценить время обучения.

### Команда

Запустите PPO с теми же параметрами, что планируете для ночи, но с **`--dry-run-seconds 120`** (или 60). Через указанное время обучение остановится и в консоль выведутся:

- **steps/sec** — число шагов окружения в секунду (сумма по всем env за rollout).
- **updates/hour** — число PPO-обновлений в час.
- Оценка **max_total_steps за 8 часов** при этой скорости.

Пример:

```bash
python train_ppo.py --bc-checkpoint checkpoints/bc/best.pt --num-envs 2 --dry-run-seconds 120
```

Или с конфигом ночного запуска:

```bash
python train_ppo.py --config configs/night_training.json --dry-run-seconds 120
```

### Как использовать результат

1. Запишите **steps/sec** и **updates/hour** из вывода `[DRY RUN]`.
2. Для ночного запуска (например 8 часов):  
   `max_total_steps ≈ steps_per_sec * 3600 * 8` (скрипт сам выведет подсказку).
3. Если используете супервизор с `max_total_steps` в конфиге — подставьте это значение.
4. Альтернатива: задать только время работы (например через супервизор или timeout), тогда лимит шагов не обязателен.

Бенчмарк **только окружения** (без PPO, один env): `python benchmark_env.py --steps 500` — см. **docs/CAPTURE.md**.

## 7.5. Как флаги влияют в реальном времени

**Почему флаги сразу срабатывают во время запуска и обучения?** Потому что они участвуют в каждом шаге rollout: окружение и тренировочный цикл читают эти параметры при инициализации и применяют их на каждом step.

### Источник действий: PPO, не BC

- **Во время обучения** агент ходит по действиям **PPO-политики** (ActorCritic). BC используется только при инициализации (`--bc-checkpoint`): копируются веса encoder (и при non-recurrent — actor). После старта обучение идёт только по PPO.
- На каждом шаге: `action = model.get_action(obs, deterministic=False)` → PPO выбирает действие.

### Novelty reward

- **Где:** в окружении, в политике наград (`RewardPolicy`). Конфиг `reward_config.novelty_reward` (0.35 по умолчанию) читается при создании env.
- **Когда:** на каждом `env.step(action)` после захвата кадра policy считает reward. Если хэш кадра (комнаты) не в `seen_rooms` и стабилен K кадров — добавляется `+novelty_reward`.
- **Как учитывается:** через reward → GAE → PPO-обновление. Политика учится чаще выбирать действия, ведущие в новые комнаты.

### Stuck-nudge (RIGHT/LEFT/UP/DOWN при застревании)

- **Где:** в `train_ppo.py`, в rollout loop.
- **Когда:** env возвращает `info["reward_stuck_event"] == True` (агент застрял N шагов без смены комнаты). Тогда `stuck_nudge_remaining[i] = stuck_nudge_steps` (20).
- **Как учитывается:** на следующих N шагах **действие PPO игнорируется** и подставляется последовательность RIGHT → LEFT → UP → DOWN. Это временная «подмена» вывода политики для выхода из застревания.
- **Почему сразу:** детекция stuck идёт внутри env (по хэшам и diff кадров). Как только событие зафиксировано — в том же rollout вступает в силу nudge.

### Recurrent (LSTM)

- **Где:** в модели (`ppo_model.py`). При `--recurrent` между encoder и головами вставлен LSTM.
- **Когда:** на каждом шаге `model.get_action(obs, hidden=(h,c))` получает скрытое состояние, возвращает новое. При `done=True` скрытое состояние обнуляется.
- **Как учитывается:** LSTM хранит контекст по траектории эпизода; политика принимает решения с учётом прошлых кадров. При `recurrent=True` добавляется ~1.5M параметров.

### Supervisor → train_ppo

- `train_supervisor.py` читает `configs/night_training.json`, строит `argv` и запускает `train_ppo.py` с флагами (`--novelty-reward`, `--stuck-nudge-steps`, `--recurrent`). Все параметры действуют с первого шага обучения.

## 8. Отладка

| Симптом | Что проверить |
|--------|----------------|
| NaN в loss | Уменьшить lr; проверить масштаб наград (REWARD.md); убедиться, что в reward нет деления на ноль. |
| Энтропия падает к нулю | Увеличить --entropy-coef; при необходимости entropy schedule. |
| Explained variance отрицательная или очень низкая | Проверить value_loss; уменьшить lr или value_loss_coef; проверить масштаб returns. |
| Unique rooms не растут | Увеличить novelty_reward; проверить stuck_penalty и max_episode_steps; убедиться, что энтропия не нулевая. |
| Много смертей | Подправить death_penalty и баланс с pickup/novelty (REWARD.md). |
| Много stuck_events | Подправить stuck_no_room_change_steps, stuck_frame_diff_threshold (REWARD.md). |

Подробнее по компонентам наград и масштабам: **`docs/REWARD.md`**.
